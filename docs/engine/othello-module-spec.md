# 오셀로 모듈 명세

## 목적
이 문서는 엔진 구현 시 핵심 모듈 경계를 고정해 책임이 섞이지 않도록 하기 위한 기준이다.

## 권장 모듈

### `board`
- 보드 상태 표현
- 좌표 범위 확인
- 칸 조회와 갱신
- 돌 개수 계산 또는 계산 지원

### `move-validator`
- 현재 플레이어 기준 유효 수 판정
- 특정 좌표에서 뒤집히는 방향과 좌표 목록 계산
- 유효 수 전체 목록 계산

### `game-engine`
- 초기 게임 생성
- 수 적용
- 턴 전환
- 패스 처리
- 종료 여부와 승패 계산

### `types`
- `CellState`, `Player`, `Position`, `GameStatus`, `MoveResult` 같은 공통 타입 정의

## 모듈 경계 원칙
- `board`는 턴 규칙이나 승패 규칙을 판단하지 않는다.
- `move-validator`는 보드를 직접 영구 변경하지 않는다.
- `game-engine`만 게임 진행 상태를 변경할 수 있다.
- 공통 타입은 순환 의존 없이 공유 가능해야 한다.

## 필수 인터페이스 성질
- 유효 수 계산은 동일 입력에 대해 항상 동일 결과를 반환해야 한다.
- 수 적용 API는 성공과 실패를 명확히 구분해야 한다.
- 실패 시 상태 불변이 보장되어야 한다.
- 종료 여부와 승패 판정은 현재 상태에서 재계산 가능해야 한다.

## 품질 기준
- 모듈 간 책임이 겹치지 않는다.
- 테스트가 모듈 단위와 통합 단위로 분리 가능하다.
- 학습 파이프라인에서 재사용할 수 있도록 보드와 게임 상태 직렬화가 가능해야 한다.

## 직렬화 최소 기준
직렬화 가능한 게임 상태는 최소한 아래 필드를 재구성할 수 있어야 한다.

- 보드 64칸 상태
- 현재 플레이어
- 종료 여부
- 종료 시 승패 결과 또는 무승부 정보

선택적으로 아래 정보를 포함할 수 있다.

- 마지막 수
- 수순 기록
- 캐시된 돌 개수
