# AGENTS.md

## 목적
이 저장소의 에이전트는 오셀로 학습 파이프라인 프로젝트를 문서 기준으로 일관되게 구현, 검토, 검증하기 위한 작업 단위로 동작한다.

이 문서는 역할별 에이전트보다 우선하는 전역 공통 규칙을 정의한다.

## 디렉터리 역할
- `docs/`: 프로젝트 기준 문서. 규칙, 모듈 경계, 인수 테스트, 학습/서빙 계약을 정의한다.
- `specs/`: 구현 전 상세 명세 문서. 언어별 인터페이스, 데이터 스키마, 프로젝트 구조, 저장 전략처럼 실제 코드 작성 전에 고정해야 하는 설계를 정의한다.
- `agents/`: 역할별 전문 에이전트 설명서. 책임 범위, 주요 입력 문서, 품질 기준을 정의한다.
- `tasks/`: 구체 작업 지시서. 구현 대상, 선행 입력, 완료 조건을 정의한다.
- `reviews/`: 리뷰 결과물. 검토 결과와 수정 우선순위를 기록하며, 가능하면 원본 폴더 구조를 그대로 미러링한다.
- `prompts/`: 반복 사용 템플릿. 구현/리뷰 에이전트에 재사용할 프롬프트 골격을 둔다.

## 작업 기본 순서
1. 먼저 관련 `tasks/*.md`를 읽어 작업 목표와 완료 조건을 확인한다.
2. 이어서 작업이 참조하는 `docs/*.md`, `specs/*.md`, `agents/*.md`를 읽어 기준과 역할 제약을 확인한다.
3. 구현 또는 리뷰를 수행한다.
4. 리뷰 작업이면 결과를 `reviews/` 아래에 기록한다.
5. 사용자가 명시적으로 요청하지 않은 범위의 코드나 문서는 수정하지 않는다.

## 리뷰 경로 규칙
- 리뷰 결과는 가능하면 검토 대상의 상대 경로를 `reviews/` 아래에 그대로 복제해 저장한다.
- 예시
  - `docs/othello-rules.md` 리뷰 결과: `reviews/docs/othello-rules.md`
  - `agents/review-agent.md` 리뷰 결과: `reviews/agents/review-agent.md`
  - `src/engine/board.ts` 리뷰 결과: `reviews/src/engine/board.ts.md` 또는 프로젝트에서 합의한 동등한 규칙
- 특정 task가 별도 출력 경로를 강제하면 그 task를 우선한다.

## 수정 허용 범위
- 사용자가 작업을 지시한 범위 안에서만 파일을 수정한다.
- 기준 문서 수정은 사용자가 명시적으로 문서 수정을 요청했을 때만 허용한다.
- 리뷰 전용 작업에서는 구현 코드를 수정하지 않는다.
- 기존 문서와 충돌하는 새 규칙을 임의로 추가하지 않는다. 필요한 경우 충돌 사실을 리뷰에 남긴다.

## 문서 우선순위
1. 사용자의 현재 요청
2. 루트 `AGENTS.md`
3. 관련 `tasks/*.md`
4. 관련 `agents/*.md`
5. 관련 `specs/*.md`
6. 관련 `docs/*.md`

동일 수준 문서끼리 충돌하면 더 작업에 가까운 문서를 우선하고, 해결되지 않으면 충돌 내용을 기록한 뒤 사용자 확인을 요청한다.

## 리뷰 원칙
- 감상이 아니라 기준 문서 대비 차이를 지적한다.
- 반드시 재현 가능한 근거를 든다.
- 사소한 스타일보다 정답성, 안정성, 설명 가능성을 우선한다.
- 문제를 쓰는 데서 끝내지 말고 왜 문제인지와 어떤 기준을 어겼는지 연결한다.
- 가능하면 우선순위를 구분한다.
  - 필수 수정
  - 권장 수정
  - 있으면 좋은 개선

## 완료 기준
- 작업 대상과 참조 문서가 명확히 연결되어 있다.
- 산출물이 지정된 디렉터리에 생성되거나 갱신되어 있다.
- 리뷰 작업이면 결과가 `reviews/` 아래에 정리되어 있고, 별도 지시가 없으면 원본 폴더 구조를 따라간다.
- 구현 작업이면 관련 기준 문서와 완료 조건을 충족한다.
